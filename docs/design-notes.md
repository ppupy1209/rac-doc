# 설계 결정 기록 (나중에 글로 정리할 것)

포트폴리오/README에 풀어 쓸 만한 설계 판단들을 모아둔다.

## 1. 캐시를 어디에 둘 것인가: 로컬(Caffeine) vs 분산(Redis)

한 앱 안에 성격이 다른 두 캐시가 있어서 저장 위치를 다르게 뒀다.

- **질의 응답 캐시 → Redis (분산)**: 최종 답변은 만드는 데 8~15초로 비싸고, 유저가 어느 인스턴스에 붙든 재사용하고 싶고, 앱 재시작에도 유지되면 좋다. 그래서 외부 분산 저장소가 이득이다.
- **임베딩 캐시 → Caffeine (로컬)**: 임베딩(글→벡터)은 자주 불리는 저지연 경로다. 인메모리는 나노초 접근이고 직렬화 비용이 없다. 게다가 잃어버려도 같은 글이면 언제든 똑같이 재생성 가능하고 재계산 비용도 낮다.

핵심 메시지: "무조건 분산 캐시"가 아니라 **캐시의 역할(생성 비용, 공유 이득, 수명 요구)에 따라 로컬/분산을 고른다.** 임베딩이 유료 API거나 인스턴스 간 공유 이득이 커지면 그때는 Redis로 옮기는 게 맞다.

글로 쓸 때 구성 아이디어: "왜 이 캐시는 로컬, 저 캐시는 분산인가?"를 질문으로 던지고, 비교 표 + 판단 기준(비용/공유/수명)으로 답하는 흐름.

## 2. 벡터 검색: 왜 인메모리 인덱스로 했나, 어떻게 확장하나

study ③에서 다룸. 측정: 20,002 청크에서 검색 6,458ms(DB 전수 스캔)에서 25ms(인메모리 인덱스)로, 약 256배.

### 문제 (baseline)
검색마다 전체 청크를 MySQL에서 읽고 임베딩 JSON을 파싱해 코사인을 계산하는 O(N) 전수 스캔. 데이터가 늘수록 선형으로 느려진다.

### 왜 인메모리 인덱스로 했나
- 목적: "매 검색마다 DB 전체 읽기 + 파싱"을 없애는 최적화 원리를 가장 단순하게 보여주기 위함.
- 장점: 네트워크 홉이 없어 극저지연. 데이터가 한 인스턴스 메모리에 들어오고 갱신이 배치성일 때 잘 맞는다. 실무 라이브러리 FAISS / HNSW / Lucene도 인메모리 계열이다.
- 한계(정직하게): 인스턴스마다 사본을 들고 있어야 하고(메모리 중복), 데이터가 바뀌면 갱신이 필요하다(추가는 증분, 삭제는 재빌드).

### 중요한 구분: 캐시용 Redis 와 벡터 검색용은 다르다
- 우리 Redis는 "질문 -> 답변" 답변 캐시(KV)다. 정확히 맞는 키로 꺼내는 것.
- 벡터 검색은 "가장 비슷한 것 찾기"라 KV GET으로 안 된다. 임베딩을 그냥 Redis에 넣고 다 꺼내 비교하면 O(N) 스캔 + 네트워크라 오히려 나빠진다.
- 벡터를 공유 저장소에 두려면 "벡터 인덱스" 기능이 있어야 한다: RediSearch(Redis 벡터 모듈), pgvector, 전용 벡터 DB.

### 확장 경로 (언제, 무엇으로)
- 멀티 인스턴스가 같은 최신 인덱스를 봐야 하거나, 데이터가 한 대 메모리를 넘거나, 실시간 갱신이 잦아지면 공유 벡터 저장소로 이동.
- 후보: pgvector(Postgres), Milvus / Qdrant / Weaviate / Pinecone, Elasticsearch·OpenSearch kNN, Redis + RediSearch.
- 판단 기준: 데이터 크기(메모리 수용 여부), 인스턴스 수(사본 감당 여부), 갱신 빈도(정적·배치 vs 실시간), 지연 요구(극저지연 vs 네트워크 홉 허용).

### 글로 쓸 때 구성
"왜 인메모리로 했나? 멀티 인스턴스는?"을 질문으로 던지고, (1) 원리 증명은 인메모리로, (2) 스케일 시엔 pgvector·전용 벡터 DB로, (3) 단 캐시용 KV Redis와 벡터 인덱스는 다르다는 점, 이 셋을 답하는 흐름.

## 3. B1 딥다이브 — DB 커밋과 벡터 인덱스의 정합성 (진행 중)

### 문제 정의
`DocumentService.create()`는 청크를 저장하면서 `vectorIndex.add()`로 인메모리 인덱스에 즉시 반영한다.
그런데 인덱스는 트랜잭션 참여자가 아니다 — 커밋 전에 반영되는 "트랜잭션 경계 밖 부수효과"라서,
롤백이 일어나면 DB에는 없는 청크가 인덱스에 남는다(유령 엔트리). 유령이 검색되면 본문 조회가 실패해
빈 content가 RAG 컨텍스트로 흘러든다. 인기글 스터디에서 다룬 "DB 커밋과 Kafka 발행의 원자성"과 동일한 문제 구조.

### 재현 실측 (2026-07-06, `GhostIndexTest`)
- 방법: Testcontainers(MySQL 8.4) 위 `@SpringBootTest`. 목 `EmbeddingClient`가 3청크 문서의 1·2번째는 정상 벡터,
  3번째에서 예외 → `create()`의 `@Transactional` 전체 롤백.
- 결과: document·chunk 테이블 **0건**(롤백 정상 동작), 벡터 인덱스에는 **유령 엔트리 2건 잔류** (`expected: <0> but was: <2>`).
- 테스트는 "롤백 후 인덱스 0건"이라는 원하는 불변식을 단언 → 현 구조에서 **의도된 red**. 수정(Step 3) 후 green 전환이 곧 증명.
- 측정 환경: 윈도우 11, Docker Desktop(Engine 29.6.1), Temurin 21, Spring Boot 3.3.5, Testcontainers 1.21.3.
  재현: `./gradlew test --tests "*GhostIndexTest"` (Docker 필요).

### 개념 정리 — 인덱스는 캐시인가? (2026-07-06 논의)
- `InMemoryVectorIndex`는 캐시라기보다 **파생 데이터(구체화 뷰)**: 캐시는 자주 쓰는 것만 부분 보관 + key 조회 + miss 시 원본 폴백으로 자기 수렴하지만, 이 인덱스는 전량 보유 + 최근접 탐색 + **miss 개념이 없다**. miss가 없으니 어긋나도 스스로 교정될 계기가 없고, 그래서 명시적 동기화(B1)가 필요하다. stale 캐시는 TTL로 죽지만 유령 인덱스는 rebuild 전까지 영생 — B1이 캐시 문제보다 어려운 이유.
- 멀티 인스턴스 관점: 지금 구조는 A 인스턴스에서 등록해도 B·C 인스턴스 인덱스에 알릴 방법이 없다 — 즉 **롤백 없이 평상시에도 정합성이 깨진다**. B1에서 만드는 동기화 구조는 공유 벡터 저장소(pgvector·Qdrant 등)로 가도 "DB 커밋 ↔ 벡터 저장소 반영" 문제로 그대로 이식된다(네트워크 너머라 오히려 더 어려움) — 이 딥다이브가 장난감 문제가 아닌 근거.

### 결정 — 공유 벡터 저장소를 지금 도입하지 않는 이유 (2026-07-06)
- **B1의 대체재가 아니다**: 공유 저장소로 가도 MySQL(원본)↔벡터 저장소(파생)의 이중 쓰기라 유령 문제가 동일하게, 부분 실패·타임아웃·재시도 중복까지 얹혀 더 어렵게 남는다. B1의 동기화 패턴을 그대로 필요로 하는 별도 이행 과제 → 순서상 B1이 먼저.
- 유일한 구조적 소멸책은 원본과 **같은 트랜잭션 저장소**에 파생 데이터를 넣는 것(Postgres+pgvector 단일화 — 한 트랜잭션이라 유령 원천 불가). 우아하지만 기각: DB 전환 비용 + study ③("MySQL엔 벡터 기능이 없어 앱에서 최적화", 256배 측정)의 서사 무효화.
- **도입 트리거 미충족**: 단일 인스턴스(배포 계획도 EC2 단일), 20k 청크 = 수십 MB, 갱신 배치성, 지연은 인메모리가 최강(25ms). 지금 해결할 문제가 없다.
- **도입 트리거(하나라도 오면 이행, 백로그 B6)**: ① 멀티 인스턴스 실배포 ② 데이터가 한 인스턴스 메모리 초과 ③ 실시간 갱신 빈도 증가. "왜 아직 안 썼나" 문서 자체가 포트폴리오 재료(D안 Kafka와 같은 서사 구조).
- **후속 갱신 (2026-07-07)**: 목표 기업 채용 공고가 "벡터 DB 기반 RAG"를 명시 → **채용 요건이 네 번째 트리거로 현실화**되어 B6를 C1(B2 다음)로 승격. 보류의 논리(B1 패턴 먼저, B2 하네스로 이행 검증)는 그대로 유지된 채 시점만 확정 — 상황이 바뀌면 결정을 재평가한다는 원칙의 실례.

### 결정 — 동기화 방식: C(Outbox+relay) 메인, B는 비교·E는 보정 관점 (2026-07-06 Step 2 결론)
- **A(현행 동기 갱신)**: 기각. `GhostIndexTest`가 기각 사유(롤백 시 유령 2건)를 실측으로 증명.
- **B(@TransactionalEventListener AFTER_COMMIT)**: 커밋 후에만 `add()` → 유령은 소멸. 남는 구멍은 "커밋 성공 후 리스너 실행 전 크래시 → 반영 누락". **단 우리 인덱스는 파생 데이터라 재시작 시 `rebuild()`가 전량 복구** → B의 유실은 "재시작 전까지의 검색 누락"으로 격하됨. 이게 Kafka 발행(시스템 밖으로 나가 유실=진짜 유실)과 정합성 요구가 다른 지점(E 관점). → **비교 대상으로 문서화**.
- **C(Outbox+relay)**: 트랜잭션 안에서 outbox 테이블에 이벤트를 원자적으로 기록 → 롤백되면 이벤트도 롤백(유령 원천 차단). relay가 폴링하며 커밋된 이벤트를 인덱스에 반영·멱등 처리. **메인 채택**. 근거 ① 인기글 스터디의 2단 발행(Outbox) 패턴을 자기 제품 문제에 전이하는 게 이 딥다이브의 서사 핵심 ② "relay kill→살리기→유실 0건"이라는 측정 가능한 장애 시나리오 확보 ③ C1(외부 벡터 저장소, 재시작-rebuild 가정이 약해짐)에서 그대로 필요해지는 구조라 이식 자산.
- **D(Kafka)**: 단일 인스턴스에 브로커는 과함 → "왜 안 썼나"를 문서화(MQTT 선택 이유와 같은 서사).
- **E(주기 diff 보정)**: 단독으론 지연이 커 부족하지만, "Outbox조차 relay가 밀리거나 이벤트 소실 시 최종 수렴 안전망이 필요"하다는 관점으로 결합 — 파생 데이터의 본질(언제든 DB에서 재구성 가능)을 살리는 층.
- **삭제·재빌드 축은 분리**: 세대(generation) 인덱스 스왑은 Step 3 후반에서 별도로 (create/add 경로 = 유령 문제 먼저).
- 면접 질문 매핑: Q2(AFTER_COMMIT 한계→Outbox 보완) = B vs C 비교, Q3(멱등성) = relay 재처리 안전, Q4(Kafka 필요 시점) = C1/멀티인스턴스 트리거.

### 결정 — 삭제 경로: outbox 통일 + 증분 제거 + AtomicReference 세대 스왑 (2026-07-07 Step 3-6)
- **코드 분석으로 ROADMAP 전제 정정**: 기존 `rebuild()`는 이미 새 리스트를 다 만든 뒤 `entries` 참조를 한 번에 교체(volatile) → **검색이 빈 인덱스를 보는 창은 원래 없었다**. 즉 "삭제 시 검색 불가 시간 = 재빌드 시간"은 사실이 아니었음. 정직한 before→after: **delete() 응답 지연(동기 전체 재빌드 = DB 전 청크 재읽기·재파싱) → ~0(이벤트 1건 기록)**, 검색 다운타임은 원래도 ~0.
- **결정**: 삭제도 create와 동일하게 **outbox로 통일**. `delete()`가 문서 삭제와 같은 트랜잭션에서 `DOCUMENT_DELETED` 이벤트를 기록(인덱스는 트랜잭션 안에서 안 건드림 → create에서 고친 것과 **대칭인 유령** 차단). relay가 `removeDocument(documentId)`로 해당 문서 엔트리만 증분 제거(DB 재읽기 없음, 없는 문서 제거는 no-op이라 멱등).
- **AtomicReference 세대 스왑**: 인덱스 내부 `volatile List<Entry>` → `AtomicReference<List<Entry>>` + `updateAndGet`(CAS). add/remove/rebuild가 동시에 일어나도 원자적 교체(무중단). ROADMAP의 "AtomicReference 원자 교체"를 문자 그대로 구현. (현 단일-relay-스레드에선 경합이 없지만, 개념 증명 + 향후 안전.)
- **V3 마이그레이션**: 삭제 이벤트는 청크 단위가 아니므로 `chunk_id`를 nullable로.
- **트레이드오프(감수)**: 삭제가 eventual(≤ 폴링 주기)이 됨 → 그 창에 삭제 문서가 빈 내용으로 잠깐 검색될 수 있으나 자가 치유(다음 relay·재시작 rebuild). 단일 인스턴스 관리자 작업엔 무해하고 create와 일관. 즉시성 필요 시 afterCommit 동기 제거가 대안이나, create에서 기각한 방식이라 일관성상 outbox 유지.

### 측정 — relay 반영 지연, 그리고 "측정 방법이 결과를 만든다" (2026-07-07 Step 3-7)
- **무엇을 쟀나**: Outbox+relay가 새로 도입한 유일한 비용 = "커밋→검색 반영" 폴링 지연. e2e(임베딩 6초에 묻힘) 대신 이것만 격리(`IndexReflectionLatencyTest`: 실제 스케줄러 on, poll=200ms, 목 임베딩).
- **자기 발견한 방법론 함정**: 1차 측정은 "반영 직후 바로 다음 create()"라 커밋이 늘 폴링 직후에 정렬 → 지연이 항상 ~T(평균 219.7ms). 이건 최악 정렬이지 평균이 아니다. 실제 업로드는 폴링 시계와 독립이므로 **create() 전 랜덤 지터(0~T)**로 균등 분포를 만들어 재측정.
- **실측(지터 적용, 20회)**: min 29 / mean 126.7 / max 359ms(첫 시행 워밍업, 이후 ≤211). 평균이 이론값 T/2=100보다 약간 큰 이유 = `@Scheduled(fixedDelay)`는 "완료 후 200ms"라 실효 주기 = 200ms + relay 처리(~20-40ms). → **모델 검증: 평균≈실효T/2, 최대≈실효T**.
- **해석/트레이드오프**: 작은 T = 신선한 인덱스 ↔ 폴링 부하↑. 프로덕션 T=1000ms면 평균 ~500ms·최대 ~1000ms. 즉시성이 정말 필요하면 이벤트 드리븐이지만 단일 인스턴스엔 Kafka 과함(D안 서사와 연결).

  | 설정 | 기대 mean | 무엇을 보여주나 |
  |---|---|---|
  | 기본 (poll=200, 지터 O) | ~127ms | 실제 평균 ≈ 폴링 주기의 절반 |
  | poll=1000 (`@SpringBootTest` 프로퍼티 + `POLL_INTERVAL_MS` 둘 다 1000) | ~500ms | 반영 지연 ∝ 폴링 주기 |
  | 지터 제거 (create() 전 `Thread.sleep(jitter…)` 주석) | ~220ms | **측정 방법이 숫자를 좌우**(폴링 직후 정렬 → 최악치) |

- **글감**: "벤치마크를 잘못 설계하면 원하는 답이 나온다" — 측정 방법(커밋 타이밍)이 숫자를 좌우한 실례. 정직한 측정 = 편향을 의심하고 통제하는 것.
- **teardown 소음 처리**: 실제 스케줄러가 켜진 컨텍스트가 캐시로 남아 컨테이너 종료 후 폴링→`Communications link failure`를 로그로 뿜음(테스트는 통과, 로그만 시끄러움). `@DirtiesContext(AFTER_CLASS)`로 클래스 종료 시 컨텍스트=스케줄러를 닫아 해결.

### 측정 — hit rate@K 기준선, 그리고 "@1이 낮은 게 진짜 신호" (2026-07-07 Step B2-3)
- **셋업**: golden set(HR/총무 8문서 + answerable 30) 적재 → relay 반영 → 각 질문 임베딩(실제 nomic-embed-text)해 top 8 검색 → 기대 문서(slug→id)가 상위 K에 있나. 채팅 LLM 없이 결정적. `./gradlew evalTest`(Ollama 필요), 기본 test에선 `@Tag("eval")`로 제외.
- **기준선(청크 500·overlap 50·topK 검색)**: @1=36.7% / @2=56.7% / **@4=93.3%** / @8=100%.
- **해석**: @1이 낮고 @4에서 급등하는 게 핵심 신호 — 질문마다 여러 문서의 청크가 상위를 다투므로 top-1은 자주 다른 문서지만, top-4쯤이면 기대 문서가 거의 포함된다. 즉 이 코퍼스에서 **topK=4 기본값은 타당**하고 @8은 +6.7%뿐(비용 대비 이득 적음). B2-5 실험 매트릭스에서 청크 크기·topK를 흔들어 이 곡선이 어떻게 변하는지 본다.
- **주의(정직)**: 8문서 소규모 코퍼스라 @4가 원래 높다. 난이도별(@4 easy 88.9%·medium 92.3%·hard 100%) 역전은 오답이 2건뿐이라 노이즈. 코퍼스를 키우면 절대값은 내려갈 것 — 중요한 건 절대값이 아니라 "무엇을 바꾸면 오르나"의 상대 비교.

### 측정 — 청크 크기 실험 매트릭스, 그리고 "실험이 개선을 못 만든 것도 결과" (2026-07-07 Step B2-5)
- **선행**: `Chunker`의 하드코딩 500/50을 설정값으로 추출(기본값 유지). 크기를 바꿔 실험 가능하게.
- **매트릭스**(청크 크기 × topK, `ChunkSizeMatrixEvalTest`):

  | chunk size | chunks | @2 | @4 | @8 |
  |---|---|---|---|---|
  | 200 | 12 | 50.0% | 73.3% | 96.7% |
  | 400 | 8 | 56.7% | 93.3% | 100% |
  | 800 | 8 | 56.7% | 93.3% | 100% |

- **발견**: 200자는 @4 73.3%로 하락 — 문서(각 300~450자)가 12청크로 파편화돼 정답 내용이 여러 청크로 흩어지고, 짧은 청크가 상위 슬롯을 나눠 가짐. 400 이상은 문서 1개=청크 1개라 동일(93.3%).
- **정직한 결론**: **청크 500·topK 4 유지**(실험이 현 기본값을 검증). 그리고 "청크 크기를 바꿔 hit rate를 올릴 여지가 이 코퍼스엔 없다"는 것 자체가 결과 — 개선이 안 나온 실험도 정직하게 보고한다. 원인은 문서가 짧아 400/500/800이 구분 안 됨. **더 날카로운 튜닝엔 긴 문서가 필요**(후속 개선 후보).
- **면접 포인트**: "청크 크기는 항상 작을수록/클수록 좋은가?" → 아니오. 문서 길이 대비 적정선이 있고, 문서보다 작게 쪼개면 파편화로 오히려 하락. 측정으로 그 경계를 봤다.

### 결정 — 생성 품질(환각) 판정: 문자열 매칭 채택, LLM-judge는 대안으로 명시 (2026-07-07 Step B2-4)
- **재는 것**: 검색(B2-3)이 아니라 **LLM이 답을 어떻게 쓰나**. ① 환각률(답 없는 질문에 지어냄) ② 오거부율(답 있는데 잘못 "모르겠습니다"). 이 둘이 프롬프트의 소심↔대담 트레이드오프를 드러냄.
- **채택 = 문자열 매칭**: 응답에 "모르겠습니다" 포함 여부로 판정. 우리 프롬프트가 no-context 시 정확히 그 문구를 내도록 설계돼(Phase A에서 확인) 잘 맞고, **결정적·재현 가능**. B2 원칙(가능한 한 결정적으로)에 부합.
- **명시한 대안(채택 안 함)**:
  - **LLM-as-judge**: 다른 LLM에게 "이 답변이 모른다고 했나 / 정답인가"를 물어 판정. 표현 변형("정보가 없습니다", "확인 어렵습니다")에 유연하지만 ① 더 느림 ② 비결정적 ③ 판정자도 틀림(판정 품질을 또 검증해야 함). 표현 다양성이 문제가 되면 그때 도입.
  - **답변 정답성(answer correctness)**: 답변을 `expectedAnswer`와 대조해 "정답을 실제로 맞혔나"까지 재는 것. 문자열 비교로는 취약(LLM이 "15일" vs 기대 "입사 1년 후 15일")해 사실상 LLM-judge 필요 → **후속 과제**로 남김.
  - **인용 정확도**: 답변 sources가 정답 문서를 가리키는 비율. 그런데 sources = topK 검색결과라 **hit rate와 동일 신호** → 독립 지표로 별도 측정하지 않음(원하면 답변 도달 건에서 부수 집계).
- **한계(정직)**: LLM이라 비결정적 → 실행마다 환각률이 조금씩 다름. 그래서 CI에 안 넣고 로컬 주기 실행(`@Tag("eval")`). 프롬프트를 바꿔 환각률이 내려가는지가 이 지표의 쓸모.

### 측정 — 환각률 기준선 65%, "1회 확인은 거짓 안심" (2026-07-07 Step B2-4 실측)
- **실측(llama3.2:3b, `HallucinationEvalTest`)**: 환각률 **65%**(답 없는 20문항 중 13개를 지어냄) / 오거부율 **0%**(답 있는 30문항 중 0개 잘못 거부).
- **해석 — 모델이 "너무 대담"**: 뭐든 답하려 들어 없는 것도 지어내고(환각↑), 대신 있는 건 절대 안 놓침(오거부 0). 소심↔대담 축에서 극단적으로 대담 쪽. 3B 소형 모델이라 "컨텍스트에 없으면 모른다고 해" 지시를 약하게 따름.
- **핵심 교훈(글감)**: Phase A에서 문서 밖 질문 1개로 "모르겠습니다 확인 → 환각 억제 OK"라고 적었는데, **20문항을 체계적으로 재니 65%가 환각**이었다. **단발 수동 확인은 거짓 안심을 준다** — 이게 golden set(체계적 평가)이 필요한 바로 그 이유. 포트폴리오 서사의 정점.
- **개선 레버**: (1) 프롬프트 강화("컨텍스트에 명시된 것만, 조금이라도 불확실하면 모르겠습니다") (2) 더 큰 모델 (3) 검색 임계값(유사도 낮으면 컨텍스트 없음 처리). B2 "측정할 숫자"대로 프롬프트 개선 전후 환각률을 비교하는 게 다음 스텝.
- **왜 환각이 나오나**: 답 없는 질문도 topK=4로 뭔가는 검색됨(가장 가까운 청크). LLM이 관련돼 보이는 컨텍스트를 받고 추론해버림. 검색이 "관련 없음"을 신호하지 못하는 것도 원인(유사도 임계값 미도입).

### 측정 — 프롬프트 강화의 트레이드오프 (2026-07-07 B2 정점, 하네스 실사용)
- **실험**: RagService 프롬프트를 "전혀 찾을 수 없을 때만 모르겠습니다"(약함) → "컨텍스트에 직접 있을 때만 답하고, 없으면 추측 말고 모르겠습니다 + 숫자·조건 지어내기 금지"(강함)로 교체 후 재측정.

  | | 환각률 (답 없음 20) | 오거부율 (답 있음 30) |
  |---|---|---|
  | before (약) | 65% (13) | 0% (0) |
  | after (강) | **20% (4)** | **26.7% (8)** |

- **핵심 = 트레이드오프를 숫자로 봤다**: 프롬프트를 엄격히 하니 환각 45%p 하락, 그러나 오거부 27%p 상승. "대담↔소심" 축에서 모델을 소심 쪽으로 민 것. **공짜 개선은 없다** — 하네스가 그 대가를 정확히 보여줬다.
- **판단은 도메인 몫(정보에 근거해)**: 사규 봇은 틀린 답(환각)이 "모른다"보다 위험(신뢰·책임) → 환각↓ 우선이면 강한 프롬프트가 낫다. 단 오거부 27%는 UX 저하라 중간값 탐색 여지. 어느 쪽이든 **감이 아니라 측정으로 결정**한다는 게 요점.
- **글감**: "AI 품질을 회귀 테스트한다"의 완성형 — 측정(65%) → 개선(프롬프트) → 재측정(20%, 단 오거부 27%). before/after 표 하나가 서사 전체를 압축. hit rate(검색)와 환각/오거부(생성)를 **분리 측정**했기에 이 진단이 가능(검색은 안 건드리고 생성만 움직임).
- **"둘 다 낮추기" 시도 — 프롬프트 4종 비교(2026-07-07, v3·v4는 Codex 협의. 프롬프트 전문은 `b2-prompt-experiments.md`)**:

  | 프롬프트 | 환각률 | 오거부율 |
  |---|---|---|
  | v1 약함 | 65% | 0% |
  | v2 강함(규칙만) | 20% | 27% |
  | v3 few-shot(자신있게+동의어) | 70% | 3.3% |
  | v4 구체항목 게이트+주제인접 부정예시 | 60% | 3.3% |

- **결론(핵심 발견)**: **프롬프트 문구는 트레이드오프 곡선 위를 이동시킬 뿐, 곡선 자체를 잘 못 민다.** v3·v4의 "자신 있게/동의어 허용"은 오거부는 잡았지만(3.3%) 모델을 다시 대담하게 만들어 환각↑(60~70%). 환각을 실제로 낮춘 건 v2(무딘 강한 규칙)뿐이고 그 대가가 오거부 27%. 3B 모델에선 프롬프트만으로 둘 다 낮추기 어렵다.
- **가설: 검색 쪽 유사도 임계값** — 약한 매치를 LLM 전에 NoContext(=모르겠습니다)로 처리하면 곡선을 밀 수 있지 않을까? **구현 전에 측정으로 검증**(`ScoreDistributionEvalTest`, 최상위 청크 score 분포):

  | 그룹 | min | mean | max |
  |---|---|---|---|
  | answerable (30) | 0.727 | 0.809 | 0.871 |
  | unanswerable (20) | 0.737 | 0.795 | 0.831 |

- **결과: 임계값도 안 통한다** — 두 분포가 **완전히 겹침**(unanswerable 최저 0.737 > answerable 최저 0.727, 평균도 0.809 vs 0.795). 깨끗이 가를 임계값이 없다. 원인: nomic-embed-text가 "육아휴직"(문서에 없음)을 휴가 문서에 대해 "연차"(있음)만큼 높게 매김 — **주제-인접 부재 항목을 의미상 가깝다고 봄**(bi-encoder 코사인의 한계). **구현 전에 측정해서 헛수고를 막은 사례.**
- **최종 진단 — 로컬 소형 모델의 환각 바닥**: 프롬프트도(곡선 위 이동만), 임계값도(점수 분리 안 됨) 못 넘는다. 뿌리 = (a) 임베딩이 인접-부재 항목을 분리 못 함 (b) llama3.2:3b가 "구체 답이 실제로 있나" 판단이 약함. **진짜 레버 = 더 강한 모델**(상용 임베딩/LLM) 또는 리랭커(cross-encoder). → **C2(상용 LLM 스위치)의 정량적 동기**가 여기서 나온다. B2 하네스가 그때 개선을 증명한다.
- **서사 완성**: 하네스로 (1) 문제 수치화(환각 65%) (2) 프롬프트 개선 시도→트레이드오프 실측 (3) 임계값 가설→측정으로 기각 (4) 구조적 한계 진단→다음 phase(C2)로 연결. 감이 아니라 데이터로 판단하고, **안 되는 것도 측정으로 안 된다고 말한다.**

### 부수 수확 — "CLI는 되는데 SDK만 안 될 때" 진단기 (글감)
Testcontainers가 "Could not find a valid Docker environment"(Status 400, 빈 Info)로 실패했는데 docker CLI는 정상.
네임드 파이프에 원시 HTTP를 직접 보내 계층을 벗겨보니 `/v1.32/info`→400, `/v1.44/info`→200 —

중간 계층(테스트 프레임워크→SDK→전송)을 하나씩 배제하고 프로토콜 레벨까지 내려가 진단한 사례.
해결: 테스트 JVM에 `api.version=1.44` 고정(레포에 커밋) — 상세는 ROADMAP B1 트러블슈팅 노트.

## 4. C2 딥다이브 — 상용 LLM 스위치, 그리고 무료 티어의 운영 현실 (진행 중)

### 측정 — C2-5 1차: Ollama 재기준선과 "실행 간 요동" (2026-07-12, 맥)

- **조건**: 맥(Apple Silicon, Docker Desktop = CPU 추론), 프롬프트 v4, 청크 500·topK 4, 모델 warm-up 후, 평가 실행에서만 `ASKWIKI_LLM_CALL_TIMEOUT_MS=120000`(함정 ① 참조). `HallucinationEvalTest` + 신규 `[LLM-USAGE]` 요약 출력.
- **실측**: 환각률 **75.0%**(15/20)·오거부율 **10.0%**(3/30). 같은 프롬프트의 B2 실측(2026-07-07, 윈도우 PC)은 60%/3.3% → **실행 간 요동이 20/30문항 표본에서 ±3문항 수준**. 원인 = LLM 비결정성 + 소표본 + 머신 차이. **단일 실행 수치가 아니라 구간(환각 60~75%)으로 읽어야 하고, 프로바이더 비교는 같은 날·같은 머신 기준선(75%/10%)과 해야 공정하다.** 결론은 불변: 로컬 3B의 환각 바닥은 높다.
- **사용량(C2-3 관측 배선의 실전 검증)**: calls 50·degraded 0·토큰 in 50,885/out 3,190(호출당 평균 ~1,018/~64)·지연 mean 8.8s/max 44.3s·LLM 합계 441s — 러너 전체 446s의 **99%가 LLM 시간**. Micrometer 지표가 provider 태그와 함께 러너 요약으로 나온다.

### 함정 3건 — 평가 하네스는 인프라와 싸운다 (2026-07-12, 글감)

- **① 장애격리 기본값이 평가를 죽임**: C2-4 가드(call-timeout 20s)가 콜드 모델 로드(2GB, 첫 호출 >20s)에 발동 → Degraded 폴백 → 러너 즉사. B2 측정 때(가드 도입 전)는 없던 간섭이 보호장치 도입으로 생겼다. 해결 = warm-up + 평가 실행만 타임아웃 완화(**양 프로바이더 동일 적용 = 조건 대칭**). 교훈: **운영 보호장치와 측정 하네스는 요구가 다르다 — 평가는 품질을 재는 도구지 가용성을 재는 도구가 아니다.**

## 5. C1 딥다이브 — 벡터 DB 이행: 정합성을 지키며 갈아타기 (진행 중)

### 결정 — 저장소 = Elasticsearch kNN, 인터페이스 = 자체 포트 + 어댑터 (2026-07-12 Step 1)

- **문제 재정의(착수 분석)**: 인메모리 인덱스의 한계 셋 — (1) O(N) 브루트포스: 검색이 데이터에 선형(20k 25ms → 1M ~1.2s 외삽), (2) 시작 시 전량 rebuild(기동 지연), (3) **멀티 인스턴스 정합성 결함**: relay가 outbox 이벤트를 PROCESSED로 마킹하면 **다른 인스턴스의 relay는 그 이벤트를 영영 못 본다** → 사본 인덱스들이 어긋남(다음 재시작까지). 현 구조는 단일 인스턴스 전제 — study ①(VT 2.3배)로 스케일 능력을 증명해 놓고 스케일아웃하면 검색이 깨지는 모순. (3)은 C1 착수 분석에서 새로 명문화 — 공유 저장소 이행의 가장 강한 동기.
- **저장소: 후보 5종 → ES kNN 채택 (연우님 결정)**: ① 강의 섹션 8 실습이 그대로 레퍼런스(학습 비용↓, 접목 P1) ② **백로그 B5(BM25+벡터 RRF 하이브리드)가 같은 엔진에서 자연 확장** — "벡터 이행"이 "검색 품질 로드맵"과 한 줄로 이어짐 ③ 업계 채택 신호 넓음. 대가(정직): 로컬 compose에 JVM 힙 ~1GB 컨테이너 추가, 벡터 전용 DB 대비 오버스펙. **기각 기록**: Qdrant(경량·전용이나 하이브리드 별도·실습 자료 없음), pgvector(MySQL 레포에 Postgres 추가 = DB 2개 운영 or 전면 이행 — 범위 폭발), Redis+RediSearch(§1 "캐시용 KV와 벡터 검색은 관심사가 다르다" 논지와 충돌 — 겸용 시 장애 도메인 결합).
- **인터페이스: 자체 포트 `VectorIndex` + 어댑터 (연우님 결정)**: 이행 기간 두 구현(InMemory·ES)을 병행해 벤치 A/B·B2 동등성 비교, 기존 테스트(Ghost/Relay)를 포트 계약 테스트로 재사용, 정합성 계약(chunkId 멱등 add·documentId 삭제)을 시그니처에 명시. Spring AI `VectorStore`(Document=String id+metadata)는 우리 도메인 모델(chunkId/documentId/seq + 본문 DB 하이드레이션)과 어긋나 **도메인에 직접 들이지 않고 어댑터 내부 구현 선택지로만** — 표준 추상화 vs 도메인 포트, "왜 안 썼나"가 글감(B1 D안 Kafka와 같은 서사 구조).
- **검증 계획**: B2 hit rate@4 동등성(기준 93.3%) + 20k 검색 지연(25ms 대비, ANN 전환 후) + 반영 지연(B1 126.7ms 대비) + 기동 시간(rebuild 제거). ANN(HNSW)은 근사라 재현율 손실 가능 — 하네스가 그걸 잡는 게 "회귀 없음 증명"의 핵심.

### 측정 — 이행 후 hit rate 동등성: 93.3% vs 90.0%@4, 그 1문항의 정체 (2026-07-12 Step 3-①)

- **조건**: 같은 코퍼스(8문서)·같은 질문 30개·같은 임베딩(nomic), B2 하네스 러너를 `VectorIndex` 포트 주입으로 바꿔 env 스위치(`VECTOR_INDEX_IMPL`)만으로 A/B. ES는 평가 시 `refresh-on-write=true`(색인 직후 검색의 가시성 지연 제거 — 품질 측정이므로).
- **실측**: memory @1=36.7/@2=56.7/**@4=93.3**/@8=100 (B2 기준선과 완전 일치 — 결정적 재현) vs elasticsearch @1·@2·@8 **완전 동일**, **@4만 90.0**(30문항 중 1문항, easy 1건이 4위→5위).
- **해석(중요)**: 코퍼스 8청크에 `num_candidates=100 ≥ N`이라 HNSW는 사실상 **완전 탐색 — ANN 근사 손실이 아님**. 원인은 **부동소수 코사인 계산 경로 차이**(우리: 사전 정규화+내적(double 누적) vs ES: 원본 벡터에 자체 cosine) → 근접 점수 쌍의 순위 스왑. 즉 "검색 품질 회귀"가 아니라 "동률 근처 랭킹의 수치 민감성". 단, **제품 기본 topK=4에서 관찰되는 차이**라는 사실은 그대로 기록한다(경계에 걸린 문서는 어차피 @8=100%로 상위권에 있음).
- **남은 검증(다음 세션)**: 20k 청크 벤치에서는 num_candidates < N이 되므로 **진짜 ANN 근사 구간**에 들어간다 — 그때 hit rate·지연을 다시 재야 "스케일에서의 트레이드오프"가 완성된다.

### 결정 — ES 접속은 Spring AI VectorStore가 아니라 네이티브 클라이언트로 (2026-07-12 Step 2-2)

- Step 1의 "도메인에 안 들인다"에서 한 발 더: **어댑터 내부에서도 Spring AI `ElasticsearchVectorStore`를 쓰지 않는다.** 결정적 이유 = **Spring AI 1.0.x의 `VectorStore.add(List<Document>)`는 스토어가 `EmbeddingModel`로 문서를 직접 임베딩하는 설계** — 우리는 임베딩을 업로드 시점에 만들어 MySQL에 저장(+캐싱)하고 relay는 그 저장분을 반영만 한다. VectorStore를 끼우면 relay가 청크마다 Ollama 재임베딩(중복 비용 + relay에 임베딩 의존성 결합). → `co.elastic.clients:elasticsearch-java`(Boot BOM 관리) 직접 사용, 매핑·bulk·kNN을 우리가 제어.
- 부수 이득: dense_vector 매핑·refresh 정책·삭제 쿼리를 명시 제어 — B1 계약(멱등 add = `_id`를 chunkId로, documentId 삭제 = deleteByQuery)을 ES 프리미티브에 1:1로 이식할 수 있다.

### 측정 — 20k 스케일 벤치: 지연 역전과 근사의 대가(recall), 그리고 dial (2026-07-12 Step 3-②)

- **조건**: N=20,000·768차원·질의 200·topK 4, **가우시안 랜덤 벡터**(구면 균등 분포 → 군집 없음 = HNSW 최악 조건, 그래서 recall은 **보수적 하한**). 인메모리(인프로세스 O(N))와 ES kNN(네트워크+HNSW)을 한 프로세스에서 나란히. 자립형 Testcontainers 벤치(`VectorIndexScaleBenchTest`, `@Tag("eval")`, Ollama 불필요 — 임베딩 목·벡터 합성).
- **실측**:

  | 지표 | InMemory (브루트포스) | ES kNN (HNSW) |
  |---|---|---|
  | 인덱스 구축 | 6,259 ms | 15,829 ms (HNSW 그래프 생성) |
  | 검색 지연 avg / p95 | 22.76 / 24.59 ms | **12.84 / 15.65 ms** |
  | recall@4 (nc=100) | 1.0 (정의상 정답) | **0.831** |

  recall dial (num_candidates): `4→0.386, 25→0.679, 50→0.773, 100→0.833, 200→0.879` (단조 증가).
- **발견 1 — 지연 역전이 이미 20k에서 일어났다**: 예상은 "20k에선 네트워크 홉 때문에 인메모리가 빠를 것"이었는데 **ES가 1.8배 빨랐다**(12.8 vs 22.8ms). 이유: 브루트포스는 질의당 20k×768 ≈ 1,540만 곱셈, HNSW는 수백 노드만 방문 → 네트워크·직렬화를 더해도 압도. **인메모리는 N에 선형(1M이면 ~1.1s 외삽), ES는 로그성 → 격차는 스케일에서 벌어진다.** 20k는 이미 교차점 이후.
- **발견 2 — 근사의 대가는 실재하고, 정확히 dial로 조절된다**: nc=100에서 recall@4=0.83 — 랜덤 데이터에서 정확 top-4의 ~17%를 놓친다. **하지만 이건 최악 케이스**: 랜덤 근직교 벡터는 top-4와 5·6위의 코사인이 거의 동률이라 "놓쳤다"가 사실상 동점 재배열이다(Step 3-①의 4↔5 스왑과 같은 현상). 실제 임베딩은 정답이 코사인 0.8 vs 무관 0.0으로 튄다 → recall이 훨씬 높다. **두 측정이 현실을 괄호로 묶는다**: 실제 코퍼스(3-①) recall≈완전, 랜덤 20k(3-②) recall 0.83·dial 가능.
- **발견 3 — `rebuild()`의 무제한 단일 벌크가 스케일에서 터진다(실측 OOM→수정)**: 첫 실행이 `EsVectorIndex.rebuild()`에서 `OutOfMemoryError`. 원인 = 20k 문서를 **하나의 BulkRequest에 쌓아** Jackson이 한 번에 직렬화(가우시안 float는 긴 소수라 벡터당 ~11KB → 단일 본문 200MB+, ES `http.max_content_length` 100MB도 초과). 수정 = `BULK_BATCH_SIZE=500`으로 배치 벌크(각 본문 유계). **벤치가 프로덕션 한계를 드러내 고친 사례** — 소규모 계약 테스트(N=3)는 절대 못 잡는 결함.
- **종합(정직)**: ES의 20k 지연 우위는 **기본 동작점(nc=100, recall 0.83)** 기준이다 — recall을 높이려 nc를 올리면 지연도 브루트포스 쪽으로 오른다(nc=200에서 recall 0.88). 즉 "ES가 무조건 빠르고 정확"이 아니라 **"HNSW는 지연·recall을 num_candidates로 맞바꾸는 곡선을 제공하고, 그 곡선이 스케일에서 브루트포스보다 유리해진다"**가 정확한 결론. 구축 15.8s(HNSW)는 인메모리 6.3s보다 느리지만 **rebuild는 드물고 ES는 영속**(재시작마다 6.3s 재빌드하는 인메모리와 달리 0). 지연 측정엔 양쪽 공통인 MySQL 하이드레이션(top-4)이 포함돼 델타는 순수 "스캔 vs HNSW+네트워크".

### 측정 — 반영 지연: ES refresh 계층이 127ms→879ms (2026-07-12 Step 3-③)

- **조건**: `EsReflectionLatencyTest` — B1과 동일 방법론(scheduler on, pollMs=200, 고정 시드 지터로 폴링 정렬 편향 제거), ES 프로덕션 기본 `refresh-on-write=false`. create() 커밋→검색가능(ES count/search는 refresh된 세그먼트만 봄 → size() 증가 시점=검색가능)까지 측정. 15회.
- **실측**: min 759 / **mean 879.0** / max 944 ms — 분포가 ~880ms에 촘촘. vs **B1 인메모리 mean 126.7ms**(같은 pollMs=200). **ES가 ~750ms 더한다.**
- **해석**: 반영 = 폴링 지연(0~200ms) + relay add + **다음 ES refresh까지 대기**. 분포가 좁고 높은 건 지배항이 폴링(가변 0~200)이 아니라 **`index.refresh_interval` 기본 1s**(거의 상수)이기 때문. 즉 인메모리는 반영 지연이 폴링 지배(평균≈T/2), ES는 **refresh 지배**로 성격이 바뀐다.
- **트레이드오프(dial 두 번째)**: `refresh-on-write=true`면 add()가 동기 refresh를 강제 → 반영이 B1의 폴링 지연 수준(~poll/2+add)으로 내려오지만 **매 쓰기가 refresh 비용**(색인 처리량↓). B1의 "폴링 주기 = 신선도 vs 부하" dial에 이어 **ES는 "refresh 정책 = 신선도 vs 처리량" dial을 하나 더** 얹는다. 우리 배포 기본은 처리량 우선(refresh-off) — 위키 문서 반영이 1초 늦어도 무해하므로 옳은 기본값. 즉시성이 필요하면 refresh-on 또는 refresh_interval 단축.

### 측정 — 기동 시간: ES는 재시작 재빌드 0 (2026-07-12 Step 3-④)

- 인메모리는 `@EventListener(ApplicationReadyEvent)` rebuild가 **매 기동마다 DB 전 청크를 로드·정규화** — 20k에서 **6.3s 실측**(Step 3-② BUILD), N에 선형(1M이면 분 단위). ES 모드는 `rebuildOnApplicationReady`가 `impl=memory`일 때만 돌게 가드돼 있어 **기동 시 재빌드 0**(ES 인덱스는 영속 — 프로세스 밖에 산다). 초기 1회 구축 15.8s는 최초 이행 때만 지불, 재시작마다가 아니다.
- **결론**: 스케일에서 인메모리의 숨은 비용은 검색 지연뿐 아니라 **기동 지연**(배포·오토스케일 시 매 인스턴스가 6.3s+ 웜업). 공유·영속 저장소는 이걸 0으로 만든다 — 멀티 인스턴스 정합성(문제 재정의 (3))과 함께 이행의 실질 동기.

### 측정 — ETL: 자체 Chunker vs Spring AI TokenTextSplitter, 짧은 문서에선 수렴 (2026-07-12 Step 3-⑤, 강의 접목 P1-②)

- **조건**: `EtlComparisonEvalTest`(결정적·임베딩 불필요). 골든셋 8문서를 자체 `Chunker`(문자 경계·500자)와 Spring AI `TokenTextSplitter`(토큰 경계·기본 800토큰)로 청킹, 문서당 청크 수 비교.
- **실측**: **8문서 전부 양쪽 1청크(총 8 vs 8, 수렴 8/8)**. 문서가 167~256자로 짧아 Chunker 500자·TokenTextSplitter 최소 청크 문턱(둘 다) 아래 → 문서=1청크로 collapse. 청킹 단위가 동일하니 **임베딩 단위 동일 → hit rate도 구조상 동일(93.3%)**, Ollama 재실행 불요(B2-5의 "짧은 문서는 크기를 구분 못 함"이 스플리터 **선택**에도 그대로).
- **결정 — Chunker 유지 (스플리터 교체 없음)**: 현 코퍼스에서 측정상 차이 0. 자체 Chunker는 문자 기반·단순·런타임 토크나이저 의존 0. Spring AI ETL의 실질 이득(토큰 인지 경계 = 모델 컨텍스트 존중, `DocumentReader`의 포맷 처리)은 **긴 문서에서만** 발현 — 실제로 쪼개지는 문서가 있어야 토큰 vs 문자 경계·오버랩 의미론이 갈린다. **교차 조건 기록**: 긴/다양한 포맷 문서를 넣으면 그때 TokenTextSplitter 재검토. VectorStore를 안 들인 것과 같은 논리 — 측정상 동등한데 더 무거운 표준 추상화를 지금 들일 이유가 없고, "언제 들이는가"를 명시.

## 6. C4 딥다이브 — ask-wiki를 MCP 서버로 노출: 표준 프로토콜로 검색 자산 개방 (완료 2026-07-13)

> **목표**: 정합성(B1)·품질(B2)로 검증한 검색을 "API로 붙였다"가 아니라 **표준 MCP 프로토콜로 AI 도구 생태계에 개방**한다. Claude Code/Desktop 같은 MCP 클라이언트가 위키를 직접 질의하는 데이터 소스가 됨. `spring-ai-starter-mcp-server-webmvc`(BOM 1.0.9, WebMVC SSE — 앱이 서블릿 MVC라 정합).

### 결정 — 2 툴 + 검증된 자산 재사용, 새 검색 스택 금지 (2026-07-13)

- **search_wiki**(검색만): 임베딩 → C1 `VectorIndex`(ES kNN) → 근거 청크. LLM 생성 없음 → 외부 에이전트가 원문 근거를 직접 당겨 스스로 추론.
- **ask_wiki**(풀 RAG): 검색 + 생성 답변 + 출처.
- **AskService 추출**: `/api/ask` 컨트롤러에 있던 캐시+RAG 오케스트레이션을 `ask/AskService`로 빼 컨트롤러와 MCP `ask_wiki`가 **한 경로를 공유**(캐시 이득·동작 일치). "MCP는 새 진입점일 뿐, 검색·정합성·캐시 자산은 재사용"이 서사. 컨트롤러는 동작 불변(회귀 그린).
- 버전 함정 재확인: 스타터 1.0.9 존재(Maven Central maven-metadata 직접 확인) — C2-1 교훈대로 착수 전 검증, 이번엔 트랩 없음.

### 발견 — 순환 의존성: "단위 초록 ≠ 컨텍스트 초록" (2026-07-13)

- **증상**: 새 단위테스트(AskService·WikiMcpTools·툴 등록) 전부 초록. 그런데 전체 스위트를 돌리자 **@SpringBootTest 전부** `BeanCurrentlyInCreationException`으로 컨텍스트 로드 실패.
- **원인**: 챗 모델 오토컨픽이 tool-calling 인프라(`toolCallingManager`→`toolCallbackResolver`)를 만들며 **모든 `ToolCallbackProvider` 빈을 기동 시 즉시 수집**한다. 우리 `wikiTools`(ToolCallbackProvider) → `WikiMcpTools` → `AskService` → `RagService` → `ChatModel`. 그런데 그 ChatModel이 방금 위 인프라를 만드느라 **생성 중** → 순환.

  ```
  ollamaChatModel → toolCallingManager → toolCallbackResolver
    → wikiTools → wikiMcpTools → askService → ragService → ollamaChatModel ⟲
  ```

- **런타임 영향은 0**: RagService는 옵션 없는 `Prompt`로 호출 → LLM에 이 툴들이 제시되지 않음(자기 자신을 재귀 호출할 위험 없음). 순수 **기동 배선** 문제.
- **수정**: 순환을 닫는 역방향 간선 하나만 지연 — `WikiMcpTools`의 `AskService` 주입에 `@Lazy`. ChatModel 생성 중엔 프록시만 주입돼 실제 AskService 생성을 호출 시점으로 미룸 → 순환 절단. search_wiki 경로(embedding·vectorIndex)는 ChatModel 무의존이라 즉시 주입 유지.
- **글감**: B2의 "1회 수동 확인은 거짓 안심"과 같은 계열 — **단위테스트만으론 못 잡고 전체 컨텍스트 부팅이 잡아낸 결함**. 그래서 위임 스펙에 "무거운 @SpringBootTest는 Codex가 만들지 말고 Claude가 전체 스위트로 컨텍스트 로드 충돌을 확인"을 명시했고(Codex 샌드박스는 gradle 불가), 실제로 그 단계에서 잡혔다. Spring AI에서 "ChatModel에 의존하는 도구를 `ToolCallbackProvider`로 등록"할 때의 정석 함정.

### 검증 (2026-07-13)

- 전체 스위트 **39/39 그린**(13클래스, Testcontainers MySQL 8.4 + ES 8.17.4). `@Lazy` 수정 후 전 @SpringBootTest가 MCP 스타터 포함 **풀 컨텍스트로 부팅**(컨텍스트 로드 충돌 없음 = C2-1식 트랩 재현 안 됨).
- 구현 Codex 위임·Claude 검증(diff + 컴파일 + 스위트), 순환 수정은 Claude 직접(1파일·`@Lazy` — 샌드박스가 gradle을 막아 Codex가 못 보는 유형, C1·B1과 같은 분업).
- (선택) 라이브 데모: 스택 기동 후 `/sse`에 MCP 클라이언트 연결 → `tools/list`에 `search_wiki`·`ask_wiki` 노출 확인 = W-3/W-4 글감.

## 7. C3 딥다이브 — 에이전틱 RAG: 구현은 한나절, 증명은 인프라에 막혔다 (진행 중)

> **목표**: 단발(검색 1회)을 넘어 LLM이 검색 필요 판단·질의 재작성·다중 검색을 주도하는 에이전틱 RAG. 프레임워크 = Spring AI Tool Calling(자바 생태계 일관, LangChain4j 기각 — ROADMAP "Phase C3"). "붙였다"가 아니라 B2 하네스로 개선(또는 한계)을 증명한다.

### 구현 — per-call 도구로 C4 순환 회피 (C3-1, 2026-07-14)

- `AgenticRagService`: `ChatClient` + `WikiSearchTool`(embed→vectorIndex.search) 도구. LLM이 도구를 0~N번 호출, Spring AI가 tool-call 루프를 내부 실행.
- **C4 교훈 이식**: 도구를 전역 `ToolCallbackProvider` 빈으로 등록하면 챗모델 tool-resolver와 순환(C4 §6). 그래서 **요청마다 `WikiSearchTool` 인스턴스를 `.tools(tool)`로 per-call 주입** → 순환 없음 + 요청별 소스 수집기. 전체 스위트 44 그린.

### 측정 설계 — 멀티홉 정답성

- 골든셋: 서로 다른 2개 문서의 사실이 **둘 다** 있어야 답하는 멀티홉 질문 6개(`eval/questions-multihop.json`, 연우님 사실검수 대기). 정답 = 답변에 factsA(문서1)∧factsB(문서2) 포함.
- 조건: 단발 `RagService` vs 에이전틱 `AgenticRagService`, topK∈{2,4}. topK=2는 단발이 한 검색에 문서를 덜 가져와 멀티홉에서 한쪽을 놓치게 → 에이전틱 재검색이 만회하는지 보는 조건. 하네스 `MultiHopComparisonEvalTest`(`@Tag("eval")`, env로 provider·topK·pacing 스위치).

### 발견 1 — 강한 모델 경로가 막혔다: Gemini thought_signature (글감 ⭐)

- **단일턴 RAG는 무료 Gemini(OpenAI-compat)로 완벽**했지만(C2/§4), **에이전틱(다중턴 tool loop)은 매 호출 HTTP 400**: *"Function call is missing a thought_signature in functionCall parts. This is required for tools to work correctly."*
- 원인: Gemini 3.x는 tool_call 응답에 `thought_signature`를 붙이고 **tool 결과를 되돌릴 때 그걸 그대로 echo**하길 요구한다. 그런데 **Spring AI의 OpenAI 어댑터는 Gemini 전용 `extra_content.google` 필드를 왕복에서 버린다** → 2번째 턴 400. (스모크 테스트로 확인: 1번째 턴은 tool_call을 정상 반환했음 = 단일턴 OK, 다중턴만 깨짐.)
- **의미**: OpenAI-compat 심(shim)은 **단일턴엔 충분하지만 에이전틱(다중턴 tool)엔 새는 추상화**다. C2의 "무료 Gemini로 챗만 교체"는 단발 RAG엔 옳았지만, 에이전틱엔 네이티브 SDK(Spring AI 1.1 `google-genai`) 또는 다른 프로바이더(Claude 네이티브 tool)가 필요하다. **추상화의 한계 지점을 측정이 정확히 짚었다.**

### 발견 2 — 약한 모델은 돌지만 이득이 없다: Ollama 실측 (2026-07-14)

- llama3.2:3b(네이티브 tool, thought_signature 무관)로는 에이전틱이 **에러 없이 돈다**. 하지만 멀티홉 정답성:

  | topK | 단발 | 에이전틱 | 평균 지연(단발/에이전틱) |
  |---|---|---|---|
  | 2 | 0/6 | 0/6 | 11.2s / 9.7s |
  | 4 | **2/6** | 1/6 | 8.8s / 13.0s |

- 에이전틱이 단발을 **못 이겼다**(오히려 topK4에서 2 vs 1). topK=2에서 둘 다 0 — 기대했던 "재검색으로 만회"가 **약한 모델에선 발현 안 됨**(3B가 다중검색·재작성을 제대로 안 함). 게다가 더 느림.

### 결론 (정직한 한계) — B2-5/C2 계열

- **에이전틱 RAG의 값어치는 "강하고 네이티브로 tool-calling 하는 모델"이 있어야 드러난다.** 현 무료/로컬 스택은 그 둘을 동시에 못 준다: 강한 모델(Gemini)은 OpenAI-compat 다중턴 tool이 막히고, 되는 모델(로컬 3B)은 약하다.
- C3-1의 정직한 결과 = **"구현했고, 현 스택에선 개선을 증명 못 함"**(B2-5 "실험이 개선을 못 만든 것도 결과"의 계열). **다음 레버**: Claude 네이티브 tool(Anthropic 키) 또는 Spring AI 1.1 native `google-genai` SDK. 코퍼스 쪽 레버(멀티홉·distractor 확장)는 강한 모델이 붙은 뒤라야 의미가 있음.

### 결정 — C3-2 라우팅: 단일턴 분류로 C3-1의 벽을 우회, AskService seam 재사용 (2026-07-14 착수)

- **왜 이게 무료로 실측 가능한 첫 C3 스텝인가**: C3-1은 *다중턴 tool 루프*라 강한 모델(Gemini OpenAI-compat)이 `thought_signature` echo 누락으로 400에 막혔다(위 발견1). C3-2의 질문 분류는 **단일턴 Structured Output**(`ChatClient.entity()`)이라 같은 Gemini가 그대로 동작한다(C2에서 단일턴 RAG는 완벽). → 추상화의 한계 지점(단일턴 OK / 다중턴 X)을 C3-1·C3-2 두 스텝이 **정확히 가른다** = 글감.
- **seam 재사용 = C4 추출의 배당**: 라우팅을 C4가 뽑은 `AskService`(캐시→RAG 오케스트레이션)에 넣으면 `/api/ask`와 MCP `ask_wiki`가 **공짜로 함께 라우팅**된다. 관심사(분류)를 새 `routing` 패키지로 분리하되 진입점은 하나로 모은다.
- **정합성 우선 기본값**: 라우팅은 분류 LLM 호출을 앞단에 **추가**한다. 그래서 `askwiki.routing.enabled` **기본 off** — default-on이면 결정적 스위트(Ollama 없이 도는 44개)가 분류 호출 때문에 Ollama를 요구하게 된다. off 기본 + **fail-open**(classify 예외/타임아웃→WIKI)으로 **기존 동작을 불변**으로 둔다. 캐시는 라우팅보다 앞에 있어 반복 질문은 분류 비용 0.
- **결과 타입 최소 리플**: `RagResult`는 sealed + exhaustive switch 2곳(`AskController`·`WikiMcpTools`)뿐 → `Clarify(message)` 1개만 신설(컴파일러가 두 switch 누락을 강제). CHITCHAT은 `Answered(message, [])` 재사용(빈 sources가 "검색 없이 답함"의 신호). 되묻기·즉답 텍스트는 분류 호출이 함께 생성(1콜, 되묻기가 질문 맞춤형이라 캔드보다 유용).
- **범위 = 코어만**(연우님 확정): 분류 정확도(golden set·confusion)는 라이브 LLM이 필요 → **C3-4로 분리**. 구현→검증→측정 순서 유지.
- **구현·검증 완료 (2026-07-14)**: 신설 `routing/`(3) + 수정 5(`RagResult`에 `Clarify`, `AskService` 라우팅, `AskController`·`WikiMcpTools` switch arm, `application.yml` 게이트) + 결정적 단위테스트(라우팅 4분기·캐시 우선·fail-open). **전체 49 그린**(44→+5), 풀 컨텍스트 부팅 OK(새 배선에 C4식 순환 없음). Codex 위임·Claude 검증. **글감 — sealed의 배당**: `RagResult.Clarify` 추가가 exhaustive switch 4곳 중 스코프에 안 넣은 eval 테스트 2곳(`Hallucination`·`MultiHop`)에서 컴파일 에러로 드러났다 = 위임 스펙의 누락을 **컴파일러가 잡아준** 사례(C4 "단위 초록 ≠ 컨텍스트 초록"의 컴파일 판). *(라이브 분류 정확도 측정은 C3-4.)*

### 리뷰 — 위임 산출물은 스펙만큼만 견고하다 (2026-07-14, C3-2 Codex 리뷰)

- **판정**: 스펙 준수 100%(지정 파일만 수정·커밋 안 함·docs 미접촉), 캐시 단락을 라우팅 **앞에** 두는 순서까지 정확. 실행 품질은 좋았다.
- **결함 1건 → 즉시 수정**: 비-WIKI 분류인데 LLM이 `message`를 빠뜨리면(`{"type":"CHITCHAT"}`) `Answered(null, [])`/`Clarify(null)`이 그대로 나갔다 — 크래시는 아니고 `"answer": null`. Codex는 `route`·`type` null은 막았는데 `message`에서 멈췄다. → **`normalize()` 순수 함수로 분리**해 "**이행 불가한 분류 = 분류 실패**"로 취급, 기존 fail-open WIKI로 흡수(캔드 문구를 새로 만들지 않고 모듈의 기존 철학을 확장). `ChatClient.entity()` 성공 경로는 목으로 못 구동하므로 **순수 함수 분리가 곧 테스트 전략** — `AgenticRagService.mapResult`와 같은 이유·같은 패턴.
- **이월 2건 → C3-4**(라우팅을 실제 켜는 단계): 분류 호출이 `LlmCallGuard`(세마포어·타임아웃)와 `LlmMetrics`(토큰·비용)를 **둘 다 우회**한다. RAG 호출은 C2-4·C2-3으로 감싸 놨는데 그 앞에 붙는 분류 호출만 무방비·비가시.
- **글감 — 두 종류의 누락**: 같은 위임에서 누락이 두 번 났는데 성질이 정반대였다. `Clarify` switch 누락은 **컴파일러가 즉시 잡았다**(sealed의 배당). 반면 가드·지표 누락은 **아무도 안 잡는다** — 컴파일도 테스트도 통과하고, 라우팅을 켜는 순간에야 hang과 과소 집계로 드러난다. → **횡단 관심사(장애 격리·관측)는 타입 시스템이 지켜주지 않으므로 위임 스펙에 명시적으로 써야 한다.** "AI에게 위임할 때 무엇을 적어야 하는가"의 구체적 사례(W-3/W-4 글감).

### 결정 — C3-3 멀티턴: 문제는 "LLM의 맥락"이 아니라 "검색의 맥락" (2026-07-14 착수)

- **핵심 통찰 (⭐ 글감)**: 멀티턴 챗봇을 RAG에 붙일 때 흔한 오해가 "ChatMemory를 넣으면 끝"이다. 하지만 Spring AI의 ChatMemory 어드바이저는 **채팅 프롬프트에 히스토리를 넣어줄 뿐**이고, RAG는 LLM보다 **먼저 임베딩·검색**을 한다. "그럼 반차는?"을 그대로 임베딩하면 *휴가* 문서가 안 잡힌다 — **LLM은 맥락을 알아도 검색이 모른다.** 그래서 본질은 메모리가 아니라 **질문 재작성(condensation)**: 히스토리 + follow-up → **독립형 질문** → 임베딩·검색. 메모리는 그 재작성의 입력일 뿐이다.
- **재작성은 단일턴** → C3-2와 같은 구조로 C3-1의 다중턴 tool 벽(`thought_signature`)을 비켜간다. C3의 세 스텝 중 **둘(분류·재작성)이 단일턴이라 무료 Gemini로 되고, 하나(에이전틱)만 막혔다**는 게 이 Phase의 지형도.
- **캐시 키를 standalone으로 (버그 예방)**: 현 `QueryCache`는 **질문 텍스트**로 키를 만든다(§1). 멀티턴이 붙는 순간 "그럼 반차는?"이 대화 A와 B에서 같은 키가 되어 **엉뚱한 답을 반환**한다 — 멀티턴이 기존 캐시를 *오염*시키는 구조적 버그. standalone 질문은 문맥 독립적이라 키로 안전하고, **부수 효과로 히트율이 오른다**(다른 표현의 follow-up이 같은 standalone으로 수렴). 한 결정이 버그를 막고 성능도 올린 사례.
- **첫 턴은 재작성 생략**: history가 비면 standalone = question → 단일턴 경로는 추가 LLM 호출 0으로 **완전 무회귀**. 턴 2부터만 1콜. 대가: 턴 2부터는 캐시 히트여도 재작성 비용을 낸다(문맥 의존 문자열로는 캐시 키를 만들 수 없으니 불가피).
- **생성 프롬프트엔 히스토리를 넣지 않는다**: 재작성으로 질문이 자기완결적이 됐으므로 생성은 **검색된 컨텍스트에만** 근거하게 둔다. 히스토리를 생성 프롬프트에 섞으면 LLM이 컨텍스트 대신 히스토리로 답할 여지가 생긴다 = B2가 65%까지 측정한 그 환각. **멀티턴을 붙이면서 환각 억제 성질을 지키는 선택.**
- **저장소 = 자체 `RedisChatMemoryRepository`** (연우님 결정): 인메모리(오토컨픽 기본)는 재시작·멀티 인스턴스에 대화가 유실된다 — **C1에서 인메모리 벡터 인덱스의 멀티 인스턴스 결함을 직접 진단해 놓고 같은 실수를 반복할 이유가 없다**(교훈의 선적용). Redis는 QueryCache로 이미 있다. 리스크였던 `Message` 다형 직렬화는 **인터페이스를 직접 직렬화하지 않고 `{type,text}` record로 내려썼다 되살리는** 방식으로 회피.
- **오토컨픽 배당**: `ChatMemoryAutoConfiguration`의 `chatMemoryRepository()`·`chatMemory()`가 **둘 다 `@ConditionalOnMissingBean`**이고 후자는 주입된 repository로 `MessageWindowChatMemory`를 빌드한다 → **repository 빈 하나만 갈아끼우면** 윈도잉·조립은 프레임워크가 해준다. C2-1(openai 스타터가 audio/image까지 오토컨픽해 컨텍스트를 무너뜨림)과 정반대로, **여기선 오토컨픽이 이득**이었다. "스타터를 믿지 말고 매번 실체를 확인하라"는 규칙이 이번엔 좋은 소식을 줬다.

### 검수 — 골든셋 독립 감사: 사실 오류 0건, 그러나 서사가 미끄러지고 있었다 (2026-07-14) ⭐

  - **위임처 선택 (판단)**: 관례상 위임은 Codex지만 이번엔 **일반 에이전트**를 썼다. 근거: 직전 두 위임에서 Codex의 PowerShell 툴링이 (a) 정상 UTF-8 JSON을 "invalid"로 오판하고 (b) 테스트 파일에 BOM을 붙여 javac가 거부했다 → **한국어 UTF-8을 신뢰성 있게 못 읽는 도구에 한국어 의미 감사를 맡길 수 없다.** 도구의 실증된 약점과 과제의 요구가 정면 충돌할 때 기본 위임처를 바꾼 사례.
  - **제약 (중요)**: 이미 공개된 측정치가 이 골든셋에 묶여 있으므로 **수정 금지·보고만**. 데이터를 고치면 기록된 숫자가 조용히 무효가 된다 → 변경은 사람이 판정하고 재측정해야 한다.
- **결과 — 사실 오류 0건**: `expectedAnswer` 30개, `turn1Answer` 8개, 멀티홉 사실 12개 전부 코퍼스와 일치. `expectedDocSlug`도 전부 정확. **B2 환각률의 타당성도 확인**: unanswerable 20문항 중 코퍼스로 답할 수 있는 것이 없다 = "65%는 엉뚱한 걸 재고 있지 않다". 부수 확인: `expectedAnswer`는 **어떤 하네스도 읽지 않는다**(grep) — 잠재 문서일 뿐(답변 정답성은 design-notes §3에서 이미 후속 과제로 유보).
- **진짜 수확 — 잡힌 건 사실이 아니라 서사였다 (글감 ⭐)**: 세 건 모두 데이터가 아니라 **내가 쓴 해설**의 문제였다.
  1. **이중잣대**: 분류 §의 `"규정"` 오답을 옹호하면서 동시에 골든셋에선 오답으로 채점 → "둘 다 성립할 수 없다"(위 절에서 정정).
  2. **틀린 대표 사례**: 대표로 인용한 `"그날이 휴일이면요?" — 주제어 0`이 사실이 아니었다. `휴일`은 코퍼스 전체에서 `salary.md` 단 한 줄에만 나오는 **기대 문서 고유 키** → none이 아니라 weak. 골든셋 라벨과 문서 문구를 함께 정정하고 층화를 재집계(none 50→100 / weak 75→100).
  3. **과잉 일반화**: "정확히 이론이 예측한 지점에서 개선이 났다"가 **1건(20~33pp) 폭의 마진**에 얹혀 있었고, `none` 정의는 그 그룹의 일부 항목에서 **경험적으로 반증**되고 있었다 → 층화는 방향 참고용으로 격하하고, 이 측정이 지탱하는 주장은 **전체 62.5→100 하나**로 한정.
  - **교훈**: 자기 데이터·자기 채점·자기 해설의 삼중 고리에서 새는 곳은 **숫자가 아니라 프레이밍**이다. 숫자는 재현되니 틀리기 어렵지만, "이 숫자가 무엇을 뜻하는가"는 유리한 쪽으로 미끄러진다. **정직성은 측정 설계가 아니라 리뷰 구조로 지켜진다** — B2에서 "1회 확인은 거짓 안심"을 배웠다면, 여기선 **"자기 해설은 거짓 확신"**을 배웠다.
- **각주로 남기고 넘어간 3건 (연우님 판정 2026-07-14 — 라벨이 모두 방어 가능하고 영향이 작거나 이미 끝난 Phase라 재측정 안 함)**:
  - **`u13` "지각 3회 시 급여가 삭감되나요?"**(RISK): 다른 19개는 코퍼스가 *주제 자체에* 침묵하는데, 이건 지각을 다루고 그 결과(월 3회 **초과** 시 서면 경고)를 명시한다 → *"삭감 조항은 없고 경고만 있습니다"*라는 **근거 있는 답이 가능**한데 `HallucinationEvalTest`는 "모르겠습니다" 미포함이면 환각으로 센다. 즉 이 1건은 **모델에게 불리한 방향의 오채점**일 수 있다(20문항 분모라 5pp). 프롬프트 계약(구체 대상=급여 삭감을 못 찾으면 모르겠습니다)상 라벨은 방어 가능해 유지. 부수: 전제가 off-by-one(3회 초과=4회부터).
  - **B2 `hard` 버킷**(OPINION): `q08`·`q20`·`q27`은 한 문장 축어 조회라 사실상 easy. `difficulty`는 `HitRateEvalTest`가 실제로 쓰고 §3의 "@4 easy 88.9%·medium 92.3%·hard 100%" 역전을 낳는데, 기록된 설명("표본이 작아 노이즈")보다 **"hard가 안 어려웠다"가 더 나은 설명**일 수 있다. 재라벨하면 B2 hit rate 재실행이 필요해 유보.
  - **멀티홉 `m3` 채점**(RISK): `factsB:["이월"]`이 **질문에 그대로 들어 있고**, 코퍼스는 "이월**되지 않는다**"인데 `String.contains`라 **정답과 그 부정을 구분 못 한다**. 그 외 한국어 수사 부분일치(`"15일".contains("5일")`)로 m4·m6도 과대 집계 소지. 전부 **통과 방향으로 부풀리는** 오차이고 C3-1의 단발 2/6 vs 에이전틱 1/6은 **1건 마진**이라 결론이 약해진다. C3-1은 이미 "미증명"으로 닫힌 Phase라 재측정 없이 한계로 기록. **멀티홉을 다시 돌린다면 `factsB`를 `["이월되지 않","소멸"]`로 고치는 게 선결.**

### 측정 — C3-4c 분류 정확도: 97.2% vs 41.7%, 그리고 fail-open이 약한 모델을 막아냈다 (2026-07-14)

- **설계**: 골든셋 `eval/questions-classification.json` 36문항(WIKI 15 / CHITCHAT 12 / AMBIGUOUS 9). **핵심 라벨링 원칙 — 라우터는 answerability가 아니라 intent를 분류한다**: "육아휴직은 며칠인가요?"는 코퍼스에 답이 없어도 **WIKI가 정답**(검색 후 "모르겠습니다"가 옳은 동작). 그래서 B2의 unanswerable 20문항을 CHITCHAT으로 재활용하면 안 된다 — 라벨을 새로 짰다. 측정은 `classify()`를 그대로 호출 = `normalize()`까지 포함한 **프로덕션 동작**. 하네스 `ClassificationEvalTest`(`@Tag("eval")`, 임베딩 불필요 → Ollama 없이도 실행 가능).

| 프로바이더 | 정확도 | WIKI recall | CHITCHAT recall | AMBIGUOUS recall | 평균 지연 |
|---|---|---|---|---|---|
| **Gemini 3.1-flash-lite** | **97.2%** (35/36) | 100% | 100% | 88.9% | 755ms |
| Ollama llama3.2:3b | 41.7% (15/36) | 100% | **0%** | **0%** | 2,476ms |

- **발견 1 — 강한 모델은 거의 완벽했고, 유일한 오답은 안전한 방향이었다**: Gemini의 단 1건 오답은 `"규정"` → WIKI(기대 AMBIGUOUS). 우리 라벨 기준으론 오답이지만 프롬프트가 명시한 **"애매하면 WIKI"(fail-open)** 방향, 즉 **되묻기 대신 검색**이라는 안전한 쪽으로 틀렸다. **위험한 방향인 WIKI→비-WIKI 누출은 0건** = 정당한 위키 질문이 라우팅에 가로채인 적이 없다.
  > **이 라벨은 논쟁적이다 (독립 검수 2026-07-14)**: 코퍼스 8개 중 **6개가 "X 규정" 제목**(휴가·근태·급여·경비·정보보안·비품)이라 `"규정"`은 도메인 내 키워드고, 위키 의도로 읽을 여지가 크다 — 검색이 *무의미*한 게 아니라 *변별력이 없을* 뿐이다. WIKI로 판정하면 정확도는 **100%(36/36)**가 된다. **연우님 판정 = AMBIGUOUS 유지**: 6개 규정 문서 중 무엇을 묻는지 특정 불가 → 임의의 top-4를 반환하느니 되묻는 게 나은 UX이고, 태그 정의("무엇을 묻는지 판단 불가")에도 맞는다. 기록된 97.2%는 이 판정 위에 선다.
  > **자기교정 기록(글감)**: 검수 전 이 절은 오답을 *옹호*하면서("명사 하나는 검색해 볼 만하고") 동시에 골든셋에선 *오답으로 채점*하고 있었다 — 양다리. 검수자가 "둘 다 성립할 수 없다"고 짚었다. **AI가 만든 골든셋을 AI가 채점하고 AI가 해설하면, 유리한 쪽으로 프레이밍이 미끄러진다.** 독립 검수가 잡아낸 게 사실 오류가 아니라 이 **서사의 이중잣대**였다는 게 이번 검수의 진짜 수확. 골든셋 설계 가설도 검증: 코퍼스에 답이 **없는** 육아휴직·퇴직금·회의실 예약을 전부 WIKI로 정확히 분류(intent ≠ answerability).
- **발견 2 — 약한 모델은 축퇴(degenerate) 분류기다 (⭐)**: 3B의 41.7%는 **골든셋의 WIKI 비율(15/36 = 41.7%)과 정확히 일치**한다. 즉 정확도가 아니라 **전부 WIKI로 찍은 결과**다 — CHITCHAT·AMBIGUOUS recall 0%. "안녕하세요"도 "1 더하기 1은?"도 위키 검색으로 보낸다. `calls=35`인데 `total=36`인 것도 단서: 1건은 구조화 출력 파싱이 아예 실패해 예외 경로로 fail-open됐다(지표 미기록). **"정확도 41.7%"를 그냥 낮은 점수로 읽으면 진실을 놓친다 — 분포와 대조해야 축퇴가 보인다.**
- **발견 3 — fail-open 설계가 약한 모델을 막아냈다**: 축퇴 분류기가 제품을 **깨지 않았다**. 전부 WIKI = 라우팅 끄기와 동일한 동작으로 degrade했을 뿐이다. 대가는 순수 낭비: **15,717 입력 토큰 + 콜당 2,476ms를 써서 이득 0**. → "약한 모델에선 라우팅을 켜지 마라"가 측정으로 뒷받침된 운영 지침이 된다(`askwiki.routing.enabled` 기본 off가 옳았다).
- **비용 (C3-4a `purpose` 태그의 첫 수확)**: Gemini 분류 36콜 = 입력 16,128 / 출력 1,214 토큰 → **콜당 약 448 입력 토큰**(시스템 프롬프트가 대부분). 라우팅을 켜면 캐시 미스마다 이만큼이 답변 호출 **위에 얹힌다**. C2-5의 답변 경로 실측(50문항 입력 50,885 ≈ 콜당 1,018)과 나란히 두면 **라우팅은 입력 토큰을 대략 +44% 늘린다**(다른 실행·조건이라 정밀 비교는 아니고 자릿수 감각). 이게 C3-4a에서 `purpose` 태그를 넣은 이유이자 첫 수확 — 태그가 없었으면 이 숫자를 분리할 수 없었다.

### 측정 — C3-4b 재작성 A/B: 검색이 맥락을 얻자 62.5% → 100% (2026-07-14) ⭐

**C3-3의 핵심 주장("멀티턴 RAG의 문제는 LLM의 맥락이 아니라 검색의 맥락")을 A/B로 증명했다.**

- **설계**: 골든셋 `eval/questions-multiturn.json` — 2턴 대화 8쌍(turn1 + 스크립트 고정 turn1Answer + 문맥 의존 followUp + 기대 문서 slug). 같은 followUp을 **(OFF) 원문 그대로 임베딩** vs **(ON) 재작성 후 임베딩** → hit rate 비교. 두 조건의 **유일한 차이는 재작성 단계**. `turn1Answer`를 스크립트로 고정한 건 turn1의 답변 품질이 결과에 섞이지 않게 하는 통제(C2의 "임베딩 고정"과 같은 규칙). 리깅 방지를 위해 앵커 없는 항목("얼마나 자주 바꿔야 해요?" — 기대 문서로 이어질 검색 키 0)과 앵커 약한 항목("식대는 하루 얼마예요?" — 재작성 없이도 맞을 수 있음)을 **의도적으로 섞음**(검수 정정 후 none 4 / weak 4). 하네스 `MultiTurnRewriteEvalTest`(`@Tag("eval")`).

| 조건 | @1 | @2 | **@4** | @8 |
|---|---|---|---|---|
| **OFF** (원문 그대로) — 3회 실행 모두 동일 | 12.5% | 25.0% | **62.5%** | 100% |
| ON — Ollama 3B (실행 1) | 25.0% | 37.5% | 62.5% | 100% |
| ON — Ollama 3B (실행 2) | 37.5% | 62.5% | 75.0% | 100% |
| **ON — Gemini 3.1-flash-lite** | 50.0% | 75.0% | **100.0% (8/8)** | 100% |

앵커별 @4(검수 정정 후 재집계, none 4 / weak 4): **앵커 없음 50.0% → 100.0%**, 앵커 약함 75.0% → 100.0% (Gemini).

- **발견 1 — 재작성은 실제로 검색을 고친다**: @4 62.5%→**100%**(8/8), @2는 25%→75%(3배). 실제 재작성: `"얼마나 자주 바꿔야 해요?"` → `"비밀번호는 얼마나 자주 변경해야 하나요?"`(OFF MISS→ON HIT) · `"식대는 하루 얼마예요?"` → `"국내 출장 시 하루 식대는 얼마까지 지원되나요?"`(OFF MISS→ON HIT). 개선의 방향은 이론과 맞는다(앵커 없음 +50pp > 앵커 약함 +25pp). **다만 층화 비교로 이론을 "증명"했다고 말하면 과잉이다** — n=4/4라 1건이 25pp고, 더 정직하게는 `none` 그룹의 정의("재작성 없으면 검색 불가")가 **4개 중 2개(welfare·onboarding)에서 반증**된다(재작성 없이도 @4 적중). 층화는 방향 참고용이고, **이 측정이 실제로 지탱하는 주장은 전체 62.5→100** 하나다.
- **발견 2 — 약한 모델의 재작성은 무의미하거나 해롭다**: 3B는 **재작성을 안 하고 답변을 한다**. `"그날이 휴일이면요?"` → `"반차는 며칠까지 사용할 수 있나요?"`(완전 무관, HIT→MISS로 **악화**), `"식대는 하루 얼마예요?"` → `"식대는 하루 약 70만 원이지만... support@koreadoumi.or.kr로..."`(**환각 금액 + 날조된 이메일**), 그리고 `申请`·`พน` 같은 **언어 혼입**. 두 실행에서 @4가 62.5↔75.0으로 **요동** = 그 "개선"은 노이즈다(C2-5의 "실행 간 요동" 재확인).
- **통제 검증 (⭐)**: **OFF가 3회 실행 모두 정확히 62.5%**. OFF는 챗 모델을 타지 않으므로 프로바이더를 바꿔도 반드시 같아야 하고, 실제로 같았다 → 하네스가 재작성 변수만 격리했다는 내부 증거. 측정이 스스로를 검산한 사례.
- **결론 — C3 전체의 지형도가 이걸로 완성됐다**: C3의 세 기능은 **"단일턴이냐 다중턴이냐"가 운명을 갈랐다.**
  - C3-1 에이전틱 = **다중턴 tool 루프** → 강한 모델(Gemini)이 `thought_signature`로 막힘 → **미증명**
  - C3-2 분류 = **단일턴** Structured Output → 강한 모델 가능
  - C3-3 재작성 = **단일턴** → 강한 모델 가능 → **증명(62.5%→100%)**
  같은 무료 스택인데 C3-1은 못 하고 C3-3은 했다. 차이는 모델도 예산도 아니라 **호출 형태**였다. "무엇이 막히고 무엇이 되는가"를 측정이 정확히 갈라 준 것 — B2의 "진짜 레버는 모델"에 이은 두 번째 구조적 발견.
- **종합 — 두 축이 C3를 설명한다 (C3-4b·4c 두 측정을 합쳐서)**:
  - **축 1: 호출 형태가 "강한 모델을 쓸 수 있느냐"를 정한다.** 다중턴 tool(C3-1) → OpenAI-compat 심이 `thought_signature`를 흘려 막힘. 단일턴(C3-2 분류·C3-3 재작성) → 무료 Gemini 그대로 동작.
  - **축 2: 모델 능력이 "그 기능이 값어치를 내느냐"를 정한다.** 같은 단일턴이어도 3B로는 **재작성이 노이즈**(62.5↔75.0 요동, 환각·언어 혼입)이고 **분류가 축퇴**(전부 WIKI, CHITCHAT/AMBIGUOUS recall 0%). 강한 모델이 붙자 재작성 62.5→**100%**, 분류 **97.2%**.
  - 두 축을 곱하면 C3의 결과가 전부 설명된다: C3-1은 축 1에서 탈락(강한 모델 접근 불가) → 미증명. C3-2·C3-3은 축 1을 통과해 축 2를 만족(강한 모델 확보) → 증명. **"에이전틱을 못 한 건 아이디어가 틀려서가 아니라 인프라가 막아서"**라는 C3-1의 정직한 결론이, 같은 스택에서 단일턴 두 기능이 성공함으로써 **반증 가능한 형태로 뒷받침됐다.**
- **정직한 한계**: 8문항·8문서의 작은 코퍼스이고 golden set은 Claude 초안(연우님 사실검수 대기). @8이 OFF에서도 100%라 이 코퍼스에선 K를 키우면 재작성 없이도 다 잡힌다 — **재작성의 값어치는 "상위 K를 좁게 쓸 때" 큰 것**(@1·@2의 격차가 그 증거). 실제 서비스가 topK=4를 쓰므로 @4의 62.5→100이 유효한 숫자다.
- **하네스 함정 (글감)**: 첫 실행에서 한국어 로그가 전부 깨져 정성 감사가 불가능했다 — 원인은 윈도우 테스트 JVM의 **stdout 기본 인코딩(cp949)**. 데이터는 정상이라 **숫자는 유효**했고 출력 경로만 뭉갠 것(재작성 문자열은 임베딩에 올바로 들어갔다). `evalTest`에 `stdout.encoding=UTF-8` 고정으로 해결. C2-5 "평가 하네스는 인프라와 싸운다"의 세 번째 사례 — **측정 결과보다 측정 도구가 먼저 깨진다.**

### 발견 — 이월한 부채는 복제된다 (2026-07-14, C3-4 착수)

- C3-2 리뷰에서 "분류 호출이 `LlmCallGuard`(가드)·`LlmMetrics`(지표)를 우회한다"를 찾고 **C3-4로 이월**했다. 그런데 바로 다음 스텝 C3-3이 새 LLM 호출부(`QuestionRewriter`)를 추가하면서 **똑같은 구멍을 그대로 복제**했다 — 위임 스펙에 안 적었으니 또 빠진 것이다. **이월 2건이 2개 호출 지점으로 불어났다.**
- 교훈: 횡단 관심사(장애 격리·관측)의 부채는 **이자가 붙는 게 아니라 복제된다.** 표준을 코드로 강제하지 않으면(가드를 지나지 않으면 호출할 수 없는 구조 등) 새 호출부가 생길 때마다 같은 누락이 재생산된다. sealed switch가 `Clarify` 누락을 **컴파일러로 강제**했던 것과 정확히 대비되는 지점 — 타입으로 강제되는 것만 반복 누락을 면한다.
- 그래서 C3-4는 측정 전에 **세 호출부(answer·classify·rewrite)를 모두 1급 시민으로** 만든다. `LlmMetrics`에 `purpose` 태그를 넣는 순간 "라우팅·재작성이 토큰을 얼마나 더 쓰나"가 **측정 가능한 질문**이 된다(공고 "토큰 최적화"에 정면 대응).
- **제약 발견**: Micrometer/Prometheus는 **같은 이름의 미터에 동일한 태그 키 집합**을 요구한다 → `purpose`를 새 호출부에만 붙이면 레지스트리가 깨진다. 기존 answer 경로까지 한 번에 붙여야 하는 이유(부분 적용이 불가능한 스키마 변경). 다행히 기존 Grafana 5패널은 전부 `sum(...) by (provider, ...)` 형태라 **라벨 추가에 하위호환**이었다.

### 발견 — 골든셋 8쌍은 100%인데, 즉흥 질문 첫 시도에서 깨졌다 (2026-07-16, 미해결) ⭐

측정이 끝나고 **포트폴리오 캡처를 찍으려고 2턴 데모를 만들다** 걸렸다. 골든셋에 없는 평범한 follow-up 하나를 즉석에서 던졌는데 "모르겠습니다"가 나왔다.

- **증상**: `1턴 "연차 휴가는 며칠인가요?"` → 정답(15일·최대 25일). `2턴 "그럼 언제까지 써야 하나요?"` → **"모르겠습니다"**, 출처에 `vacation` 없음(onboarding·equipment·attendance·expense).
- **원인 = 재작성기의 과잉 특정**. 3회 재현, 매번 같은 성격:

  | | 질문 | 출처 | 답 |
  |---|---|---|---|
  | 재작성기 산출 | `연차 휴가는 **입사일로부터** 언제까지 사용해야 하나요?` | onboarding, equipment, attendance, expense | ❌ 모르겠습니다 |
  | 통제(그 한 마디만 제거) | `연차 휴가는 언제까지 사용해야 하나요?` | onboarding, equipment, **vacation**, salary | ✓ 12월 31일 소멸 |

  1턴 답변에 있던 **"입사 1년 후"**를 주워다 사용자가 묻지도 않은 **"입사일" 조건**으로 승격시킨다. 문장으로는 멀쩡한 독립형 질문이지만, 그 한 마디가 임베딩을 지배해 `vacation`을 top-4 밖으로 밀어낸다. **재작성기는 대명사를 채우랬더니 전제를 발명했다.**
- **fail-open이 아니라 "잘 돌아서 틀렸다"**: `llm_calls{purpose="rewrite"}=1`, `llm_degraded` 없음 → 가드·예외 경로가 아니라 **성공한 호출의 내용이 틀린 것**. C3-4a가 붙인 `purpose` 태그가 "재작성이 돌긴 했나"를 5초 만에 갈라 줬다 — **관측 투자의 두 번째 수확**(첫째는 분류 비용 +44%).
- **62.5→100%를 뒤집지 않는다.** 그 8쌍 골든셋 밖의 질문이고, 골든셋 재실행 결과는 그대로다. 하지만 **"골든셋 100% = 기능이 된다"가 아니라는 것**은 뒤집는다.
- **교훈 — 골든셋은 측정이지 보증이 아니다.** n=8짜리 셋에서 100%를 받은 기능이 **즉흥 질문 첫 타석에서 깨졌다.** 골든셋은 내가 고른 질문이고, 내가 고른 질문은 내가 상상한 실패만 담는다. 검수(2026-07-14)가 "자기 데이터·자기 채점·자기 해설은 프레이밍이 샌다"를 잡았다면, 이건 **"자기 골든셋은 커버리지가 샌다"** — 같은 병의 다른 얼굴이다. 역설적으로 이 결함을 찾아낸 건 새 측정이 아니라 **제품을 실제로 써 본 것**(캡처용 데모)이었다.
- **다음 레버(미실행)**: 재작성 프롬프트에 "추측한 조건·수식어를 덧붙이지 말고 대명사만 채워라"를 추가. **단 프롬프트를 건드리면 멀티턴 eval 재실행이 따라오고**(62.5→100이 이 프롬프트의 숫자다), 검증하려면 골든셋을 즉흥 follow-up으로 넓히는 게 선결이다 — 그래야 같은 실수를 또 안 놓친다.
