# 설계 결정 기록 (나중에 글로 정리할 것)

포트폴리오/README에 풀어 쓸 만한 설계 판단들을 모아둔다.

## 1. 캐시를 어디에 둘 것인가: 로컬(Caffeine) vs 분산(Redis)

한 앱 안에 성격이 다른 두 캐시가 있어서 저장 위치를 다르게 뒀다.

- **질의 응답 캐시 → Redis (분산)**: 최종 답변은 만드는 데 8~15초로 비싸고, 유저가 어느 인스턴스에 붙든 재사용하고 싶고, 앱 재시작에도 유지되면 좋다. 그래서 외부 분산 저장소가 이득이다.
- **임베딩 캐시 → Caffeine (로컬)**: 임베딩(글→벡터)은 자주 불리는 저지연 경로다. 인메모리는 나노초 접근이고 직렬화 비용이 없다. 게다가 잃어버려도 같은 글이면 언제든 똑같이 재생성 가능하고 재계산 비용도 낮다.

핵심 메시지: "무조건 분산 캐시"가 아니라 **캐시의 역할(생성 비용, 공유 이득, 수명 요구)에 따라 로컬/분산을 고른다.** 임베딩이 유료 API거나 인스턴스 간 공유 이득이 커지면 그때는 Redis로 옮기는 게 맞다.

글로 쓸 때 구성 아이디어: "왜 이 캐시는 로컬, 저 캐시는 분산인가?"를 질문으로 던지고, 비교 표 + 판단 기준(비용/공유/수명)으로 답하는 흐름.

## 2. 벡터 검색: 왜 인메모리 인덱스로 했나, 어떻게 확장하나

study ③에서 다룸. 측정: 20,002 청크에서 검색 6,458ms(DB 전수 스캔)에서 25ms(인메모리 인덱스)로, 약 256배.

### 문제 (baseline)
검색마다 전체 청크를 MySQL에서 읽고 임베딩 JSON을 파싱해 코사인을 계산하는 O(N) 전수 스캔. 데이터가 늘수록 선형으로 느려진다.

### 왜 인메모리 인덱스로 했나
- 목적: "매 검색마다 DB 전체 읽기 + 파싱"을 없애는 최적화 원리를 가장 단순하게 보여주기 위함.
- 장점: 네트워크 홉이 없어 극저지연. 데이터가 한 인스턴스 메모리에 들어오고 갱신이 배치성일 때 잘 맞는다. 실무 라이브러리 FAISS / HNSW / Lucene도 인메모리 계열이다.
- 한계(정직하게): 인스턴스마다 사본을 들고 있어야 하고(메모리 중복), 데이터가 바뀌면 갱신이 필요하다(추가는 증분, 삭제는 재빌드).

### 중요한 구분: 캐시용 Redis 와 벡터 검색용은 다르다
- 우리 Redis는 "질문 -> 답변" 답변 캐시(KV)다. 정확히 맞는 키로 꺼내는 것.
- 벡터 검색은 "가장 비슷한 것 찾기"라 KV GET으로 안 된다. 임베딩을 그냥 Redis에 넣고 다 꺼내 비교하면 O(N) 스캔 + 네트워크라 오히려 나빠진다.
- 벡터를 공유 저장소에 두려면 "벡터 인덱스" 기능이 있어야 한다: RediSearch(Redis 벡터 모듈), pgvector, 전용 벡터 DB.

### 확장 경로 (언제, 무엇으로)
- 멀티 인스턴스가 같은 최신 인덱스를 봐야 하거나, 데이터가 한 대 메모리를 넘거나, 실시간 갱신이 잦아지면 공유 벡터 저장소로 이동.
- 후보: pgvector(Postgres), Milvus / Qdrant / Weaviate / Pinecone, Elasticsearch·OpenSearch kNN, Redis + RediSearch.
- 판단 기준: 데이터 크기(메모리 수용 여부), 인스턴스 수(사본 감당 여부), 갱신 빈도(정적·배치 vs 실시간), 지연 요구(극저지연 vs 네트워크 홉 허용).

### 글로 쓸 때 구성
"왜 인메모리로 했나? 멀티 인스턴스는?"을 질문으로 던지고, (1) 원리 증명은 인메모리로, (2) 스케일 시엔 pgvector·전용 벡터 DB로, (3) 단 캐시용 KV Redis와 벡터 인덱스는 다르다는 점, 이 셋을 답하는 흐름.

## 3. B1 딥다이브 — DB 커밋과 벡터 인덱스의 정합성 (진행 중)

### 문제 정의
`DocumentService.create()`는 청크를 저장하면서 `vectorIndex.add()`로 인메모리 인덱스에 즉시 반영한다.
그런데 인덱스는 트랜잭션 참여자가 아니다 — 커밋 전에 반영되는 "트랜잭션 경계 밖 부수효과"라서,
롤백이 일어나면 DB에는 없는 청크가 인덱스에 남는다(유령 엔트리). 유령이 검색되면 본문 조회가 실패해
빈 content가 RAG 컨텍스트로 흘러든다. 인기글 스터디에서 다룬 "DB 커밋과 Kafka 발행의 원자성"과 동일한 문제 구조.

### 재현 실측 (2026-07-06, `GhostIndexTest`)
- 방법: Testcontainers(MySQL 8.4) 위 `@SpringBootTest`. 목 `EmbeddingClient`가 3청크 문서의 1·2번째는 정상 벡터,
  3번째에서 예외 → `create()`의 `@Transactional` 전체 롤백.
- 결과: document·chunk 테이블 **0건**(롤백 정상 동작), 벡터 인덱스에는 **유령 엔트리 2건 잔류** (`expected: <0> but was: <2>`).
- 테스트는 "롤백 후 인덱스 0건"이라는 원하는 불변식을 단언 → 현 구조에서 **의도된 red**. 수정(Step 3) 후 green 전환이 곧 증명.
- 측정 환경: 윈도우 11, Docker Desktop(Engine 29.6.1), Temurin 21, Spring Boot 3.3.5, Testcontainers 1.21.3.
  재현: `./gradlew test --tests "*GhostIndexTest"` (Docker 필요).

### 부수 수확 — "CLI는 되는데 SDK만 안 될 때" 진단기 (글감)
Testcontainers가 "Could not find a valid Docker environment"(Status 400, 빈 Info)로 실패했는데 docker CLI는 정상.
네임드 파이프에 원시 HTTP를 직접 보내 계층을 벗겨보니 `/v1.32/info`→400, `/v1.44/info`→200 —
docker-java의 구식 기본 API 버전(1.32)을 Docker Engine 29(최소 1.40)가 거부하는 것이 원인이었다.
중간 계층(테스트 프레임워크→SDK→전송)을 하나씩 배제하고 프로토콜 레벨까지 내려가 진단한 사례.
해결: 테스트 JVM에 `api.version=1.44` 고정(레포에 커밋) — 상세는 ROADMAP B1 트러블슈팅 노트.
